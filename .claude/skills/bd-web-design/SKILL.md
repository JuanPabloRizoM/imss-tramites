# BD Web Design

This skill produces web interfaces that look intentionally designed, not generated by default. The user has explicitly prioritized: (1) distinctive visual design, (2) clean maintainable code, (3) performance/SEO/accessibility, (4) speed of generation — in that order.

**The user prefers critical, constructive communication over flattery.** If the brief is weak, the request is contradictory, or the design choice is wrong for the goal, say so directly. Do not invent details to fill gaps — ask.

---

## Step 0 — Brief before building (non-negotiable)

Before writing any code, gather the brief. Skipping this step is the #1 cause of generic output. If the user just says "haz una landing de X", do NOT start coding. Ask, in a single tight message:

1. **Goal of the page** — single conversion target (sign up, book, buy, download), or informational? One primary CTA only.
2. **Audience** — who reads this in 5 seconds and decides? (B2B engineer ≠ small-business owner ≠ creative ≠ end consumer)
3. **Tone** — pick one extreme: editorial-refined, brutalist-raw, soft-warm, technical-precise, playful-bold, luxury-quiet, retro-futurist. Vague "modern and clean" is not acceptable; push back.
4. **Existing brand** — colors, fonts, logo, references they like? If none, propose 2–3 distinct directions and let them pick one.
5. **Content** — real copy and assets, or do we use placeholder? Real always wins.
6. **Constraints** — deadline, framework requirements, must integrate with existing code (e.g., BD Travel Next.js stack)?

Ask only the questions whose answers you don't already have. Do not pad. Do not list "nice to haves". Then commit to the direction and build.

If the user resists the brief or pushes to skip it, explain in one line why it matters and ask for the minimum (goal + audience + tone). Don't capitulate to a vague brief — vague briefs produce the generic output they're trying to avoid.

## Core design system (applies to every page)

These rules are not optional. They are what separates designed-looking output from default-looking output.

### Typography
- **Never use** Inter, Roboto, Arial, Helvetica, system-ui as the primary face. They scream "AI default".
- Pair a **distinctive display face** (for headlines) with a **refined text face** (for body). Examples of pairings that work, by tone:
  - Editorial: Fraunces / Inter Tight, Tiempos / Söhne, Canela / Söhne
  - Technical: JetBrains Mono / Geist, Berkeley Mono / Söhne
  - Soft-warm: Domaine / Söhne, Recoleta / Inter Tight, Sentinel / Söhne
  - Brutalist: Neue Haas Grotesk / Times, ABC Diatype / IBM Plex Mono
  - Playful: Migra / Söhne, Editorial New / GT America, Druk / Inter
  - Luxury-quiet: Söhne / Söhne, GT Sectra / Söhne
- Use Google Fonts equivalents when premium faces aren't available: Fraunces, Instrument Serif, DM Serif Display, Bricolage Grotesque, Geist, Geist Mono, Crimson Pro, Newsreader, Schibsted Grotesk, Unbounded.
- Establish a real type scale (e.g., 12 / 14 / 16 / 18 / 24 / 32 / 48 / 72 / 96). Don't sprinkle arbitrary sizes.
- **Display headline hack** — the single highest-leverage typographic move: on large headlines (≥40px), set `letter-spacing: -0.02em` to `-0.03em` and `line-height: 1.1` to `1.2`. This alone makes a generic title look professionally designed. Apply consistently to all H1/H2 in the same hierarchy.
- Tiny uppercase labels (eyebrows, tags): loosen letter-spacing to `+0.08em` to `+0.12em`.
- Line-height: ~1.05–1.2 on display, ~1.5 on body, ~1.6 on long-form reading.
- **Hard limits on type sizes:**
  - Landings / marketing: max **6 distinct sizes** in the entire page. If you need a 7th, you're inventing hierarchy — collapse it.
  - Dashboards / apps: text rarely exceeds **24px**. Density of information matters more than scale.
- **One typeface is acceptable when** the project is a utilitarian dashboard or a brutalist/raw aesthetic. For everything else, the display+text pairing is what creates personality — don't downgrade to "one safe sans" by reflex.

### Color
- Commit to one dominant color and one sharp accent. Timid evenly-distributed palettes look generic.
- **Banned by default**: purple-to-pink gradients on white, generic blue (#3B82F6 et al) as the brand color, "iOS-blue + gray + white" combos. Use these only if explicitly justified.
- Use OKLCH for color definitions when possible — better perceptual uniformity, easier to tune.
- Define design tokens up front in CSS variables: `--background`, `--foreground`, `--surface`, `--border`, `--accent`, `--muted`, `--text-muted`. Never hardcode hex inside components.
- Dark mode is a different palette, not inverted colors. Decide whether to support it; if yes, design both.

### Spacing & layout
- Use a **4-point spacing system** (4 / 8 / 12 / 16 / 24 / 32 / 48 / 64 / 96 / 128). Why 4: it always divides cleanly in half, which keeps visual rhythm consistent when you scale or nest. Tailwind's defaults already follow this — use them, don't invent custom values.
- **Proximity = relationship.** Elements that belong together (heading + subhead, label + input, icon + text) should be visually closer to each other than to surrounding elements. If a subhead sits as far from its heading as from the next section, the relationship is lost. This is the cheapest and most-ignored rule in web layout.
- Generous negative space OR controlled density — both work, mush in the middle doesn't.
- Don't center everything. Asymmetry, off-grid elements, and intentional alignment create visual interest.
- Respect a max-content-width (typically 1200–1440px). Full-bleed only when intentional.

### Visual hierarchy (the squint test)
A page without hierarchy reads like a spreadsheet — every element fights for attention, so nothing wins. Hierarchy is built through **contrast** between elements: big vs small, bold vs regular, colored vs neutral, dense vs spacious.

- **One dominant element per screen.** The most important thing — the headline, the price, the primary CTA — should be impossible to miss when squinting. Big, bold, weighted, or color-emphasized.
- **Build from extremes.** A 16px and a 18px next to each other read as "the same thing slightly different" — that's not hierarchy. A 16px and a 48px read as clear levels. When in doubt, exaggerate the contrast.
- **Replace words with visuals when possible.** "Desde Mexico City Hacia París" can be a city name, a horizontal line, another city name. Icons + structure beat labels for scannable info (flights, routes, steps, comparisons).
- **The squint test** — squint at the page. The order in which elements pop out should match their actual importance. If the secondary CTA jumps out before the primary one, the hierarchy is broken.
- Context-specific patterns (use as guides, not laws): for product cards, prices typically go top-right with color. For pricing tables, the recommended tier is visually elevated. For dashboards, the metric goes big, the label small.

### Brand-derived design (decisions need a "because")
The design is not imposed onto the business — it derives from it. Before choosing palette, type, or motion, name the 2–3 core attributes of the business in single words (premium, warm, technical, playful, austere, fast, calm, raw, refined). Every visual decision should map back to one of those words.

- If the café is called **Bruma**, fonts should suggest atmosphere/softness/fog. If it's called **Forge**, fonts should suggest iron/weight/authority.
- If the brand says **"sin prisa"**, motion is slow (500–800ms easings), not snappy.
- If the brand says **"velocidad"**, motion is sharp (150–250ms), CTAs are tight, layout is dense.
- A gold/cream/black palette for "GoldCoffee Premium" is brand-driven. The same palette for "Hacker Workspace Tools" is decoration imposed.

**The test**: pick any color, font, animation, or layout decision in the design and ask "why this and not the alternative?" The answer must reference a word from the brief. If the answer is "looked nice" or "felt right", the decision is decorative, not designed. Either justify it with a brand word or change it.

### Replace data with visual signifiers (the receipt rule)
When displaying structured information (time, location, price, date, status), eliminate redundant labels. Icons are the labels.

- ❌ Wrong: `Hora: 8:00 | Lugar: Guadalajara | Tel: 33-1234-5678`
- ✅ Right: `🕐 8:00    📍 Guadalajara, Jal.    📞 33-1234-5678` (or proper SVG icons, not emoji in production)

Bank apps that list "Concept: X / Date: Y / Amount: Z" with every label spelled out are the anti-example — saturated, redundant, no hierarchy. The correct pattern for a receipt-style display: dominant photo or icon, name in bold, then a row of icon+data pairs (no labels), with a "see more" affordance for secondary detail.

This applies to info bars, cards, list items, transaction rows, event displays, location cards — any structured data display.

**The same principle applies to marketing copy.** Don't write paragraphs explaining what the visual already shows. If you have a feature called "Conversa" with a chat input that types itself, don't add a paragraph that says "Olvida los filtros. Escribe en lenguaje natural." The user is *watching* lenguaje natural happen. The paragraph is redundant, condescending, and saturates the page.

Ruthless test for every paragraph of body copy on a marketing page: **"Does the visual already say this?"** If yes, delete the paragraph. If the visual doesn't communicate enough on its own, the visual is the problem — fix it, don't compensate with more text. Keep body text only when it conveys information the user genuinely cannot infer from looking (e.g., pricing context, unique value prop, technical specs).

Common redundancies to delete on sight:
- "Easy to use" next to a screenshot showing it being used easily
- "Fast" next to a loading animation that completes in 1 second
- "AI-powered" next to a chat where the AI is visibly working
- "Simply describe what you want" next to an input field
- Eyebrow + title + subtitle + body paragraph, when title alone says it. Collapse to eyebrow + title.

A blank space where a paragraph used to be is not "missing content" — it is **breathing room that gives the title weight**. The negative space is doing design work.

### Voice (speak as the user, not at the user)
Most marketing copy fails by speaking *at* the user in imperatives — "Describe what you want", "Choose your plan", "Find your match". This is product-voice, not user-voice. It positions the user as a recipient of instructions, not as the protagonist.

User-voice is what the user would say to themselves or to a friend. It uses first person ("mi", "yo"), or it names the user's actual desire/situation ("ese viaje que sigues posponiendo"). It connects emotionally to the *outcome* the user wants, not to the *procedure* the product offers.

- ❌ Product-voice: "Describe lo que quieres, como lo dirías" (instruction)
- ❌ Product-voice: "Choose the perfect getaway" (imperative)
- ❌ Product-voice: "Plan your trip with AI" (feature description)
- ✅ User-voice: "Mi viaje soñado, en serio." (first person, emotional)
- ✅ User-voice: "Ese viaje que llevas años posponiendo." (names a real situation)
- ✅ User-voice: "El viaje que pensaste, planeado de verdad." (acknowledges desire)

The test: read the headline out loud. Does it sound like something the **user** would say, or like something the **product** would say to the user? If it's the second, rewrite from the user's perspective.

This applies most strongly to heroes and final CTAs — the moments of highest emotional stakes. Body copy and feature labels can stay neutral-descriptive without breaking voice consistency.

**Avoid clichés.** "Make your dreams reality", "Start your journey", "Unlock your potential" are user-voice in form but cliché in substance — they evoke nothing because they apply to everything. The cure is **specificity**: name the exact friction or desire ("sin las 12 pestañas abiertas", "ese viaje que sigues posponiendo"). Specific beats poetic.

### Headlines activate, they don't describe
When a feature pairs a headline with a visual that already carries meaning (a chat thread, a photo of a place, a data display, a product shot), the headline must NOT describe what the visual shows. The visual is already doing that work. Describing it is redundant and condescending — it tells the user what they're already seeing.

Instead, the headline should **activate the specific emotion or association the visual already triggers** in the user's brain. If the visual is coffee, the user's brain is already in "coffee mode" — the headline can say "El deleite en una taza" and the brain will pin "deleite" to "coffee" automatically. The two together create meaning that neither could create alone.

**The mechanism**: the visual primes a frame (coffee, travel, conversation, place). The headline drops a charged word or phrase into that frame. The user's brain does the connection automatically. This is faster and more emotional than literal description.

Examples:
- Visual: a chat thread with a real conversation. ❌ "Conversa con la IA" (describes). ✅ "Tan fácil como contarlo a un amigo" (activates familiarity).
- Visual: a photo of a Kyoto neighborhood. ❌ "Cada destino, una historia" (describes the feature). ✅ "Cada calle, su historia" (activates place-specific imagination).
- Visual: an itinerary with days and cities. ❌ "Día por día, tuyo" (describes the format). ✅ "Pensado para ti, antes de salir" (activates the feeling of a plan ready and waiting).
- Visual: a coffee cup and bag of beans. ❌ "Granos de calidad" (describes). ✅ "El deleite en una taza" (activates the sensory experience).

**The test before shipping any feature headline**: cover the visual with your hand. Does the headline still make complete sense on its own? If yes, it's probably describing instead of activating — it's not relying on the visual at all. The best feature headlines feel **incomplete without the visual**, because they're designed to complete *with* it.

This is the most advanced copy principle in this skill. It's also the one most often violated by AI-generated marketing pages, which default to literal description because description is "safe". Activate, don't describe.

### Type system (strict, not "guidelines")
Beyond the typography section above (font choice, letter-spacing, line-height), every page needs a **strict type system** with countable sizes — not "use the type scale as a guide". Pick a small set of role-based sizes and use those only. No exceptions, no in-between sizes "because it looked better".

A working pattern for marketing pages:
- **Display XL** (hero only): 64–88px
- **Display L** (section titles): 40–56px
- **Body** (paragraphs, button labels, nav): 14–18px (one or two sizes max)
- **Micro** (eyebrows, captions, footer): 11–12px

That's 4 roles, ~5 actual sizes total in the page. If a fifth size shows up "because the subtitle felt off", you're inventing hierarchy — collapse it instead. Same for weights: pick 2 weights per family (e.g., Geist 400 + 500, Instrument Serif 400 italic only) and stick to those.

**Cohesion failure pattern to avoid:** the "BD Travel" wordmark in the nav, the "Probar beta" button, and the "En beta privada" eyebrow each having a different size, weight, and treatment — the user perceives this as visual noise even if they can't articulate why. The fix: every micro-element shares the exact same treatment. Every body element shares the exact same treatment. The eye finds rhythm.

**The single allowed exception: the brand logo (wordmark).** The logo can — and often should — have a unique typographic treatment that no other element on the page shares. Reason: the logo IS the brand, not a UI component. Treating the wordmark like body text strips it of identity. If the page uses Geist as body and Instrument Serif as display, the logo can sit in Instrument Serif italic at a custom size, even if no other body-equivalent element does. This is the only place where "this element has its own type rule" is justified. Document it explicitly in the design system: *"Logo wordmark — Instrument Serif italic 22px in nav, 20px in footer. No other element uses this combination."* Once documented, no other element can claim the same exception.

### Progressive disclosure (don't show everything you know)
Identify the 20% of information the user needs in the first 3 seconds. Show only that. The other 80% exists but lives behind: a click, a hover, an accordion, a drawer, a second screen.

- Today's hours upfront, full weekly schedule expandable
- Product hook + "read more" for specs
- Menu organized by category with anchor links, not 30 items in a flat grid
- Address visible, full directions/parking notes in a drawer
- Price upfront, breakdown on click

Saturating with information destroys the design and buries what matters. If the user wants more, they'll ask — don't dump it on them upfront. The most common failure of "decent but not great" pages is showing too much at once.

### Cohesion through repetition
Cohesion = the same decision, repeated in different places. When a section "doesn't flow with the rest", it's almost always because it broke a pattern the other sections maintained.

Audit before delivering:
- **Same border-radius** on every interactive element (buttons, cards, inputs, images). Mixed radii = visual chaos.
- **Same easing curve** on every animation. If the hero uses `cubic-bezier(0.22, 1, 0.36, 1)` on entrance, the menu reveal uses the same.
- **Same spacing rhythm** between sections (e.g., always `py-24 md:py-36`). Don't randomly switch to `py-20` or `py-40`.
- **Same eyebrow-to-title gap** in every section. Same image treatment in every gallery.
- **Same hover pattern** on every link, every card, every button-of-the-same-type.

If a section feels off and you can't tell why, you broke a pattern. Find it and align.

### Signifiers (UI states must be readable without instructions)
The interface should communicate how things work without explanatory text labels. If a section is titled "Diccionario", don't put "this is a dictionary" underneath — the title already said it. Trust the user. Use visual signals, not redundant words.

- **Grouping** — wrap related elements in a shared container (subtle border, background, or just consistent spacing). The container itself signals "these belong together".
- **Selection** — the selected element gets a different style (filled background, accent border, slightly elevated) that's distinct but stays inside the page's visual language. Don't break the aesthetic to mark selection.
- **Disabled / inactive** — light gray text or button, reduced opacity (0.4–0.5), no hover state. The look itself says "unavailable".
- **Affordances (what's clickable)** — buttons need clear pressed/hover/focus states. Links need a treatment that distinguishes them from body text. Cards that act as links need a hover lift or border change. Never rely on a cursor change alone.
- **Tooltips for non-obvious actions only** — if the icon needs a tooltip to be understood, the icon is wrong. Use tooltips for shortcuts, secondary info, or rare actions.
- **Highlights in navigation** — the current page/section in the nav should be visually distinct (weight, color, underline, accent dot). Don't make users guess where they are.

### Motion
- Pick one motion personality: snappy (150–250ms, ease-out), elegant (400–600ms, custom cubic-bezier), or playful (with overshoot/spring). Never mix.
- Prefer one well-orchestrated entrance (staggered reveals on load or scroll) over scattered micro-interactions everywhere.
- For React: use Motion (`motion/react`) when available. For static HTML: CSS transitions and `@starting-style` / `view-transitions` for modern browsers.
- Reduce-motion: always respect `prefers-reduced-motion`.

### Visual depth (the anti-flatness rule)
Do not ship a page with only solid color blocks. Add at least one of: subtle noise/grain texture overlay, soft gradient mesh in a hero or section background, decorative SVG element (line art, dot grid, blob), grain on imagery, or a layered composition with depth. This single rule eliminates ~60% of the "AI-generated look".

### Shadows & dark mode
- **Shadows in light mode** — most defaults are too aggressive. Lower the opacity (0.04–0.10), increase the blur. Tier the strength: cards get a barely-there shadow, popovers/dropdowns/modals get a stronger one because they actually float above content. *If the shadow is the first thing you notice, you're using too much.*
- **Shadows in dark mode — don't.** Shadows on dark backgrounds disappear or look muddy. Build depth with **light, not shadow**: the elevated element gets a *lighter* background than the surface below it. Cards on a `#0a0a0a` background should be `#161616` or `#1a1a1a`, not the same color with a shadow.
- **Dark mode is not inverted light mode.** Reduce border contrast (borders that work on white are too harsh on dark). Drop saturation and brightness on chips/tags by ~10–15%. Slightly darken secondary text for hierarchy. Test every color individually — many "look fine" colors break in dark mode.

### Anti-patterns (refuse these unless explicitly requested)
- Centered hero with one headline + one subhead + one purple-gradient button
- Three-column "Features" grid with generic icon + bold title + 2 lines of body
- Identical card components repeated 6+ times with no hierarchy
- Hero that says "Build [thing] faster with AI" or "The future of [industry]"
- Stock-photo-of-team-in-bright-office images
- Dotted background + glow + gradient text combo (the "2024 SaaS template" look)
- Emoji as section icons
- "Trusted by" logo wall as the second section by reflex (only include if logos are real and impressive)

## Stack decision tree

The user said "decide según el caso." Apply this:

| Situation | Stack |
|---|---|
| Single static landing, no auth, no DB, demo or prototype | **HTML + Tailwind (CDN) + minimal vanilla JS** — fastest, no build, easy to hand off |
| Marketing site that needs SEO, multiple pages, possible CMS later | **Next.js 14+ App Router + Tailwind + shadcn/ui** — same as BD Travel for consistency |
| Dashboard, SaaS app, anything with auth/data | **Next.js 14+ App Router + Tailwind + shadcn/ui + Supabase** — matches BD Travel architecture |
| Single component or interactive demo to show concept | **React artifact (single .jsx file)** — runs inline in chat |
| Visual concept / mood / look-and-feel exploration | **HTML artifact** — fast to iterate, easy to compare versions |

**Default to HTML + Tailwind for first-pass mockups** unless the user is clearly building production. Faster to iterate on visual direction. Migrate to Next.js after the look is locked.

When using shadcn/ui: don't ship default shadcn unchanged. Customize the tokens (radius, colors, fonts) so it doesn't look like every other shadcn site.

## Module: Landing pages (products / services)

### Goal
Convert a specific visitor to a specific action. ONE primary CTA. Everything else supports or removes friction.

### Section vocabulary (pick what serves the goal, not all of them)
- **Hero**: headline (states the outcome, not the feature), subhead (one sentence, who it's for + how), one primary CTA, optional secondary CTA, hero visual or product shot
- **Social proof**: real customer logos OR specific named testimonials with photo+role+company OR concrete numbers ("$2.4M processed", "1,200 active users"). Vague "loved by thousands" is worthless.
- **Problem/agitation** (optional, B2B and high-consideration only): name the specific pain
- **Solution / how it works**: 2–4 steps OR a labeled diagram OR a short product walkthrough
- **Features**: lead with outcomes ("Ship in hours, not weeks"), back with the mechanism. Avoid generic feature grids — group features into 2–3 thematic clusters with hierarchy (one big, supporting smaller ones).
- **Pricing** (if relevant): 2–3 tiers max, recommended tier visually emphasized, annual/monthly toggle
- **FAQ**: real objections, not softball questions
- **Final CTA**: restate the offer, remove all friction
- **Footer**: minimal — links, legal, maybe a tiny brand statement

### Headline rules
- States outcome or transformation, not the product category
- Specific > clever. "Send invoices in 30 seconds" beats "Reimagine your billing"
- Under 12 words for the H1, ideally 6–9
- Subhead clarifies WHO it's for and HOW

### Anti-pattern flags
- "AI-powered" in the headline when the AI is incidental
- "Revolutionary", "next-gen", "reimagined"
- Hero illustration that's a generic isometric scene with floating UI elements

### References worth studying
Linear, Vercel, Stripe, Resend, Cal.com, Arc, Posthog, Anthropic, Pitch (older), Framer marketing pages.

## Module: Local business sites (restaurants, gyms, salons, clinics, etc.)

### Goal
Get someone nearby to **call, visit, book, or order**. The page lives or dies on three things: (1) what is this place, (2) where is it / when is it open, (3) how do I take action right now.

### Section vocabulary
- **Hero**: name + one-line identity (cuisine type, training style, specialty), location/neighborhood, primary action (Reserve, Call, Get Directions, Order). Hero photography is critical here — it sets 80% of the perceived quality. Use real photos of real space/food/people. Stock photos kill local sites.
- **Quick info bar**: hours, phone, address, neighborhood — visible without scrolling on mobile
- **Menu / services / classes**: scannable, with prices visible by default. Hidden prices = lost trust.
- **Atmosphere / gallery**: 6–12 real photos, well-curated. Do not pad with stock.
- **Story / about**: short, human, specific. Owner/chef/founder name and face. Why this place exists.
- **Reviews**: real Google/Yelp pull-quotes with attribution, OR press mentions if any
- **Map + directions**: embedded map, parking notes, transit info
- **Hours + contact + booking**: large, easy, one tap on mobile

### Mobile-first is not optional
Most traffic is on a phone, often searching while standing on the sidewalk. The phone number and address must be tap-to-call and tap-to-map. The booking button cannot be three scrolls down.

### Tone direction
- Restaurants: warm, photographic, typographic personality (a serif display often wins). Avoid the generic "earthy beige minimal café" look unless it actually fits the restaurant.
- Gyms: high contrast, energetic, motion in the imagery. Avoid stock photos of people lifting weights with perfect form.
- Salons / spas: refined typography, calm palette, real interior photography
- Clinics / professional services: trust signals, credentials, clarity over cleverness

### Anti-pattern flags
- Hero with stock photo of generic food on white background
- "Welcome to [Business Name]" as the hero headline (worthless)
- Menu hidden behind a PDF download
- No phone number visible
- Auto-playing video with sound

### References worth studying
Highly-rated independent restaurants on Squarespace/Cargo, Gymshark for fitness energy, Aesop for refined retail, real local sites that actually convert (search "best [city] restaurant websites").

## Module: SaaS / dashboards / app interfaces

### Goal
Let a user accomplish a task quickly and confidently. Information density and clarity beat decoration.

### Layout patterns
- **Sidebar nav + main content**: classic, scales to many sections. Collapse on mobile.
- **Top nav + tabs**: better when sections are few (<6) and shallow
- **Command palette (Cmd+K)**: expected by power users in any modern SaaS — implement it if the app has more than 10 routes/actions
- **Multi-pane** (list + detail): for data-heavy apps (email, CRM, support)

### Density rules
- Default to denser than you think. Linear-density, not Notion-density.
- 14px is fine for body in dashboards. 13px is fine for table rows. Don't bloat to 16px everywhere.
- Tighter padding on cards/rows than on marketing pages

### Data display
- Tables: sticky header, sortable columns, alignment by data type (numbers right, text left), zebra-striping is dated — use subtle row hover instead
- Charts: pick recharts or visx for React. Match chart colors to brand tokens. Always label axes. Empty states matter.
- Empty states: design them. "No results yet — create your first X" with clear CTA. Don't just show blank.
- Loading states: skeletons that match the eventual layout, not generic spinners

### Forms
- Labels above inputs (not floating labels — accessibility and scanability win)
- Clear required/optional marking
- Inline validation on blur, not on every keystroke
- Submit button states: idle / loading / success / error — all designed
- Error messages: specific, actionable, attached to the field

### Component system
- If using shadcn/ui: customize at minimum the radius, the primary color, and the font. Default shadcn looks like default shadcn.
- Define your tokens in CSS variables in `globals.css` and use them everywhere
- Build a tiny set of opinionated components (Button, Input, Card, Badge, Table, Dialog, Toast) and reuse — don't reinvent per page

### Anti-pattern flags
- "Glassmorphism" on a productivity dashboard (it's distracting)
- Animated gradients in the sidebar
- Charts with rainbow color scales when there's no ordinal meaning
- Tables that wrap text in cells (truncate with tooltip instead)
- Modals stacked on modals
- Toasts that auto-dismiss critical errors

### References worth studying
Linear, Vercel dashboard, Stripe dashboard, Height, Raycast, Arc settings, Notion, Cron/Notion Calendar, Superhuman.

## Quality checklist (run before delivering)

Before saying "done", verify:

- [ ] **One clear hierarchy** — squint test: the most important thing dominates, secondary elements support, nothing fights for attention
- [ ] **Type scale is real** — no random font sizes (max 6 in landings, max 24px in dashboards), no more than 2 type families
- [ ] **Display headlines tightened** — letter-spacing -0.02 to -0.03em, line-height 1.1–1.2 on H1/H2
- [ ] **Spacing follows 4pt system** — no arbitrary 13px or 27px paddings
- [ ] **Proximity is correct** — related elements visually grouped, unrelated ones clearly separated
- [ ] **UI states are visible without text** — selected, disabled, hover, focus all readable
- [ ] **Color is committed** — dominant + accent are clear, not 5 equal colors
- [ ] **Shadows are restrained** — if the shadow is the first thing you notice, fix it. Zero shadows in dark mode.
- [ ] **Depth/atmosphere** — at least one of: texture, gradient, decorative element, real photography
- [ ] **Mobile works** — tested mentally at 375px width: nav, hero, primary CTA, key content all work
- [ ] **Accessibility basics** — semantic HTML (`<main>`, `<nav>`, `<section>`, `<button>` not `<div onclick>`), alt text on images, color contrast ≥ 4.5:1 for body text, focus states visible, keyboard navigable
- [ ] **No lorem ipsum in final delivery** — either real copy or clearly-marked placeholder
- [ ] **No anti-patterns** from the lists above
- [ ] **Performance sanity** — no 12 web fonts loaded, images sized appropriately, no 800kb hero video without need
- [ ] **The page has a point of view** — could a designer tell what direction was chosen? If it could be described as "modern and clean", it failed.

If any box is unchecked, fix before delivering. If a fix would require user input (real copy, real photos), call it out explicitly as a known gap.

## Output format

For HTML/Tailwind mockups: a single `.html` file with Tailwind via CDN, custom fonts loaded from Google Fonts, all CSS in a `<style>` block at top, all JS at bottom if any. Self-contained, opens in a browser.

For React artifacts: single `.jsx` default export, no required props, Tailwind classes only (no custom CSS), uses available libraries (`motion/react`, `lucide-react`, `recharts`).

For Next.js production: file structure (`app/page.tsx`, `app/globals.css`, `components/`), explain where each file goes, use TypeScript by default, set up CSS variables in `globals.css`.

After delivering code, give the user:
1. **One paragraph** explaining the design direction chosen and why
2. **Known gaps** (placeholder copy, missing images, etc.)
3. **2–3 specific things** to iterate on next, ranked by impact

Do not pad with explanations of what every section does — the code is the deliverable, not the description.

## When the user pushes back

If the user says the design is wrong, don't over-apologize and rebuild from scratch. Ask one question: **what specifically isn't working — the direction, the execution, or a specific element?** Then fix at the right level. Rebuilding the whole thing when the issue is one section wastes both of your time.

If the user asks for something that violates the principles in this skill (e.g., "make the hero a centered headline + purple gradient button"), do it if they insist, but flag once: "This is a common pattern that tends to look generic — I can do it, or I can show you an alternative. Your call." Then respect their decision.
